## Proposition d'architecture — Traitement LIDAR

### 1. Ce qui manque dans les structures existantes

**Problème critique** : `Robot` n'a pas d'**orientation**. Impossible de convertir les points polaires LIDAR (angle, distance) en coordonnées absolues sur le plateau sans connaître l'angle du robot.

**Problème secondaire** : les dimensions du plateau sont codées en dur (`10x10` dans `isInsideMap`) et il n'y a aucune constante pour 300×200 cm.

#### C. Nouveau fichier : `includes/LidarProcessing.h` + `src/LidarProcessing.cpp`

C'est le cœur du traitement. Séparé de `vision.cpp` pour la modularité.

**Pipeline de traitement :**

```
ScanPoints (polaire, relatif au robot)
    │
    ▼
① polarToAbsolute()         ← conversion en coordonnées plateau (x_cm, y_cm)
    │                          utilise : position robot + orientation robot
    ▼
② filterOutsideMap()        ← supprime les points hors 300×200
    │
    ▼
③ clusterPoints()           ← regroupe les points proches (naïf, 1 seul cluster attendu)
    │
    ▼
④ estimateEnemyPosition()   ← centroïde du cluster → position adverse
    │
    ▼
⑤ checkZoneVisits()         ← pour chaque zone, si le robot adverse est à proximité → marquer visitée
```

**Structures à créer :**
```cpp
struct AbsolutePoint {
    double x_cm;  // position absolue sur le plateau
    double y_cm;
};

struct Cluster {
    std::vector<AbsolutePoint> points;
    AbsolutePoint centroid;  // calculé après clustering
};
```

`runLoop()` appellera simplement le pipeline de `LidarProcessing` et écrira les résultats dans le `Board` via les setters existants (`moveEnemyRobot`, `setStorageZoneState`, etc.).

### 3. Complexité des calculs (par scan, ~10 Hz)

| Étape | Complexité | Coût estimé |
|-------|-----------|-------------|
| ① Conversion polaire→cartésien | O(n) | n ≤ 8192, trivial |
| ② Filtrage hors plateau | O(n) | simple comparaison x/y |
| ③ Clustering | O(n) single-pass | après filtrage, n très réduit (~50-200 points restants) |
| ④ Centroïde | O(k) | k = taille du cluster |
| ⑤ Check zones | O(z) | z = 19 zones (8+10+1) |

**Total : < 1ms par scan** — aucun problème de performance.

#### Pourquoi un clustering naïf suffit :
- Après filtrage des bordures et du sol, il ne reste que les objets **au-dessus de 7cm et en dessous de ~17cm** (hauteur LIDAR)
- Seul le robot adverse sera dans cette catégorie (les éléments de jeu type palets sont au sol)
- Un simple regroupement par distance (tous les points à < `ENEMY_ROBOT_RADIUS_CM` les uns des autres) suffit
- Pas besoin de DBSCAN ou K-means

---

### 4. Formule de conversion polaire → absolu

```
x_abs = robot.x + (dist_mm / 10.0) * cos((robot.orientation_deg + point.angle_deg) * π/180)
y_abs = robot.y + (dist_mm / 10.0) * sin((robot.orientation_deg + point.angle_deg) * π/180)
```

Un point est dans le plateau si : `0 ≤ x_abs ≤ 300` et `0 ≤ y_abs ≤ 200`.
